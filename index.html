The Problem

The Solution

Built end-to-end real-time mocap pipeline in Unreal Engine 5. Managed mocap lab (Manus gloves, Xsens suit, Faceware) with weekly capture sessions and 2 actors. Same node supported both live mocap streaming and baked Maya animation playback.

StoryboardTo3D

Download Full Thesis (PDF)
Role: Sole Developer & Researcher
Context: MS Thesis, Drexel University
Completed: December 2025

Overview

Hand-drawn storyboards enable rapid visualization in animation and game development, but translating 2D sketches into 3D scenes requires considerable manual effort. This thesis presents StoryboardTo3D, an Unreal Engine 5.6 plugin that automates storyboard-to-3D translation using vision-language models. The system implements a multi-angle capture framework (7 viewpoints) and iterative positioning refinement with AI feedback loops.

The Problem

Translating 2D storyboards into 3D scenes requires hours of manual camera placement, object positioning, and depth interpretation. Existing research prototypes convert text-based screenplays into preliminary scenes but do not efficiently translate visual storyboard sketches into 3D layouts using production asset libraries.

The Solution

An Unreal Engine 5.6 plugin with modular AI provider abstraction supporting three backends (Claude Sonnet 4.5 Extended Thinking, ChatGPT-4o, LLaVA-13B). The system's primary value is time-shifted automation: users initiate positioning processes that run unattended—including overnight batch processing—and return to completed 3D scenes. While processing takes 4–41× longer than expert manual work (1.7 min baseline), it runs without human labor, enabling workflows for storyboard artists and indie developers who lack Unreal Engine expertise.

System Pipeline

Seven-stage iterative refinement loop: analyze → generate → capture multi-angle → AI feedback → adjust → repeat until convergence.

Voice-Controlled Robot Combat

Role: Solo Developer
Context: Drexel University Course Project

The Problem

Action games demand physical dexterity—thumbstick precision, button timing, complex input combinations. This excludes players with motor impairments from an entire genre. Most accessibility solutions are post-hoc remapping that still assumes some form of manual controller use.

The question: can voice input be a primary control method for real-time combat, not just an assistive add-on?

The Approach

Design a combat game from the ground up around voice control. No button fallback, no hybrid input—if it doesn't work with voice alone, it doesn't ship. This constraint forces design decisions that prioritize accessibility rather than retrofitting it.

Voice Architecture

Built on Windows Speech Recognition via Unity's KeywordRecognizer API. Recognition runs entirely on-device—no cloud round-trip, which keeps the input-to-action loop tight enough for real-time gameplay.

Voice command flow: recognition → dispatch → animation → AI reaction

Accessible Vocabulary Design

Seven commands total—small enough to memorize immediately, large enough for tactical depth. Each keyword is phonetically distinct to minimize recognition errors across accents and speech patterns.

Movement

"advance" · "back" · "jump"

Combat

"punch" · "kick" · "upper cut" · "block"

No homophones, no phonetic overlap. "Advance" instead of "forward" (avoids confusion with "block"). "Upper cut" as two words for clearer recognition than "uppercut."

Deliberate Pacing

Voice input is inherently slower than button presses—there's latency from utterance to recognition to action. Instead of fighting this, the design embraces it: each command locks input until the animation completes. You commit to your action.

This is the key accessibility insight: games that don't reward fast inputs don't penalize players who can't produce fast inputs. The result plays more like calling plays in football than twitch combat.

Action Commit Window Risk/Reward
kick 1.13s Quick, low commitment
block 1.49s Defensive window
punch 2.09s Standard attack
upper cut 3.08s High damage, punishable

Reactive AI Opponent

The AI runs a finite state machine that reacts to player commands. When it detects the player attacking within range, it has a 50-80% chance to block—creating the illusion of reading your voice commands.

AI state transitions based on distance and player behavior

Distance-based logic: advances when far, retreats when close, attacks within strike range. State transitions have a 0.5s buffer to prevent erratic behavior. The AI plays by the same deliberate rules as the player—readable, tactical, fair.

Accessibility Context

This project explores voice as a primary input method—removing the physical dexterity barrier that excludes players with motor impairments from most action games. No formal accessibility testing was conducted; this is design research into the viability of the approach, not a validated accessibility solution.

The design aligns with industry direction: Xbox Accessibility Guidelines, PlayStation Access controller, and games like The Last of Us Part II proving that accessible design expands audience. Voice control remains an underexplored input modality with potential to open action games to currently excluded players.

Limitations

Platform: Windows only—KeywordRecognizer wraps Windows Speech Recognition, unavailable on Mac/Linux/WebGL.

Recognition: Keyword grammar (fixed vocabulary), not continuous dictation. No confidence thresholds or misrecognition feedback implemented.

Validation: No formal latency measurement, recognition accuracy testing, or accessibility user testing conducted. This is a design prototype, not a production-validated system.

Scope: Proof-of-concept exploring voice-as-primary-input for accessible action games.
